# Deploying the Load Generator Automatically in Google Cloud

## Objective

Automate the deployment of the load generator on a GCE VM inside the same GCP
region as the GKE cluster, using Terraform for VM provisioning and Ansible for
configuration. This gives more realistic performance measurements than running
locally, since network latency between the load generator and the frontend is
minimised.

---

## Why run the load generator in the cloud?

Running Locust locally means every request travels from your machine, across the
internet, to the GCP datacenter hosting the cluster. That network latency becomes
part of every measured response time, making it impossible to isolate application
performance from network conditions.

Running the load generator on a GCE VM in the same region means:

- Requests stay within GCP's internal network
- Network latency is consistently low (sub-millisecond between VM and cluster)
- Response times reflect actual application performance, not your ISP

---

## Architecture

```
Your machine
    |
    | terraform apply + ansible-playbook
    ↓
GCE VM (europe-west3-a, e2-medium)
    |
    | HTTP requests (same VPC, low latency)
    ↓
Frontend LoadBalancer (35.242.229.106)
    |
    | internal gRPC (ClusterIP)
    ↓
CartService, CheckoutService, ProductCatalogService, ...
```

---

## Tool responsibilities

| Tool | Responsibility |
|---|---|
| Terraform | Provision VM, create firewall rules, generate Ansible inventory |
| Ansible | Install Docker, copy load generator files, build image, run container |

This separation mirrors real infrastructure-as-code practice: Terraform manages
*what exists*, Ansible manages *what runs on it*.

---

## File structure

```
infrastructure/
  terraform/
    main.tf           # VM, firewall, inventory file generation
    variables.tf      # all configurable values
    outputs.tf        # VM IP and Locust UI URL
    inventory.tpl     # template for Ansible inventory
    terraform.tfvars  # actual values (gitignored)
  ansible/
    loadgenerator/
      playbook.yml    # installs Docker, builds image, runs Locust
      inventory.ini   # generated by Terraform (gitignored)
```

---

## How Terraform and Ansible connect

Terraform generates `inventory.ini` automatically after the VM is created using
a template file (`inventory.tpl`). The template is filled with the VM's external
IP and the Locust configuration variables:

```ini
[loadgenerator]
${vm_ip} ansible_user=debian ansible_ssh_private_key_file=~/.ssh/google_compute_engine

[loadgenerator:vars]
frontend_addr=${frontend_addr}
locust_users=${users}
locust_spawn_rate=${spawn_rate}
```

This means there is no manual copy-pasting of IPs between tools. Terraform knows
the VM IP, writes it to a file, and Ansible reads it from that file.

## Steps to deploy

### 1. Authenticate

```bash
gcloud auth application-default login
```

### 2. Set up SSH key for Ansible

```bash
ssh-keygen -t ed25519 -f ~/.ssh/google_compute_engine -N ""

gcloud compute project-info add-metadata \
  --metadata-from-file ssh-keys=<(echo "debian:$(cat ~/.ssh/google_compute_engine.pub)") \
  --project=<your-project-id>
```

### 3. Configure variables

Edit `infrastructure/terraform/terraform.tfvars`:

```hcl
project_id        = "your-project-id"
region            = "europe-west3"
zone              = "europe-west3-a"
machine_type      = "e2-medium"
frontend_addr     = "<frontend-external-ip>"  # from get-frontend-ip.sh
locust_users      = 10
locust_spawn_rate = 1
```

> `frontend_addr` must be the bare IP with no `http://` prefix. The Dockerfile
> entrypoint prepends `http://` automatically.

### 4. Provision the VM

```bash
cd infrastructure/terraform
terraform init
terraform plan -out=tfplan
terraform apply tfplan
```

### 5. Install Ansible Docker collection

```bash
ansible-galaxy collection install community.docker
```

### 6. Configure and start Locust

```bash
ansible-playbook \
  -i infrastructure/ansible/loadgenerator/inventory.ini \
  infrastructure/ansible/loadgenerator/playbook.yml
```

### 7. Verify Locust is running

```bash
ssh -i ~/.ssh/google_compute_engine debian@<vm-ip> \
  "sudo docker logs locust"
```

Successful output shows requests with 0% failures and response times in the
20-100ms range.

## Headless mode

Locust runs in headless mode (`--headless` flag in the Dockerfile entrypoint),
which means no web UI is available on port 8089. All output goes to stdout,
visible via `docker logs locust`.

To enable the web UI, the Dockerfile entrypoint would need to be modified to
remove the `--headless` flag and rebuild the image. For automated load testing
headless mode is preferred — the web UI is useful for interactive exploration
but adds no value in automated runs.

---

## Tearing down

To avoid unnecessary GCP costs, destroy the VM when not running load tests:

```bash
cd infrastructure/terraform
terraform destroy
```

This removes the VM and firewall rule. The GKE cluster is managed separately
and is not affected.

---

## Design decisions

| Decision | Benefit | Trade-off |
|---|---|---|
| Same region as cluster | Low latency, realistic measurements | Slightly higher cost than a cheaper region |
| `e2-medium` machine type | Sufficient for Locust, cheapest viable option | Cannot run many parallel Locust workers |
| Terraform-generated inventory | No manual IP copy-pasting between tools | Inventory is regenerated on every apply |
| Headless Locust | Simpler, no UI overhead | Less interactive than web UI mode |
| `restart_policy: unless-stopped` | Container survives VM reboots | Must explicitly stop to change config |
